# MSA 환경의 옵저빌리티 계측 — 실무 운영 가이드

`@ObservationKeyValue` 실습 가이드의 후속 문서. 단일 서비스에서 계측을 붙여본 다음,
**마이크로서비스 10개 규모에서 이걸 어떻게 운영하는가**를 아키텍처·운영 관점으로 정리한다.

핵심 명제: **계측은 한 종류가 아니다. 성격에 따라 3개 층으로 쌓고, 층마다 책임 주체와 방법이 다르다.**
어노테이션(`@ObservationKeyValue`)은 그중 한 층(도메인 계층)일 뿐이며, MSA의 무게중심은 다른 곳에 있다.

---

## 0. 자주 하는 오해부터

> "MSA면 서비스가 10개니까 계측 어노테이션도 10번 중복된다. 공통 모듈로 빼야 하지 않나?"

이 질문은 **두 가지 다른 문제를 섞고 있다.** 분리하면 답이 명확해진다.

- **인프라 설정의 중복** (OTel 설정, 샘플링, 공통 태그) → 진짜 중복. 공통화한다.
- **도메인 태그의 분산** (`order.type`, `payment.method`) → 중복이 아니라 **분산**. 각자 둔다.
- **여러 서비스가 공유하는 횡단 계측** (공통 외부 호출 등) → 진짜 중복. convention으로 공통화한다.

| 무엇 | 중복인가 | 어디에 둘까 |
|------|:---:|------|
| OTel 설정·샘플링·공통 태그 | 진짜 중복 | 사내 starter(auto-config) 또는 OTel Agent |
| 서비스 고유 도메인 태그 (`order.type`) | 중복 아님 (분산) | 각 서비스에 어노테이션으로 |
| 여러 서비스 공유 횡단 계측 (예: 공통 인증/연계 호출) | 진짜 중복 | 공통 모듈에 convention으로 |

`@ObservationKeyValue(key="order.type")`는 그 서비스의 **도메인 지식**이다.
결제 서비스가 주문 태그를 알 필요가 없으므로 공통 모듈에 있을 이유가 없다.
"10개 서비스에 어노테이션이 흩어진다"는 건 같은 코드의 복사가 아니라, 서로 다른 도메인 태그가
각자 제자리에 있는 것 — 정상이다. (컨트롤러 매핑이 서비스마다 다른 게 중복이 아닌 것과 같다.)

---

## 1. 계측의 3층 구조

### 1층 — 자동 계측 (대부분은 코드를 안 짠다)

표준 구간(HTTP 요청/응답, DB 쿼리, Kafka produce/consume, 외부 HTTP 호출)은 자동으로 span이 된다.

- **프레임워크 내장 instrumentation** — Spring Boot + OTel 스타터만 붙이면 컨트롤러·RestClient·JDBC가
  자동 계측된다. 실습에서 `http get /orders` span이 우리가 만든 게 아니라 자동으로 나온 것이 이 경우.
- **OTel Java Agent** — 코드·의존성 변경 0. JVM 시작 시 `-javaagent`로 바이트코드를 조작해 계측을 주입.
  서비스가 수십 개인 MSA에서 선호된다(이미지/사이드카에 Agent 하나만 박으면 됨). 단, 버전 호환성
  이슈가 있고 비즈니스 의미가 있는 커스텀 태그는 못 붙인다.

> **MSA 옵저빌리티의 70~80%는 이 1층에서 끝난다.** 서비스 간 호출 흐름, 병목 위치, 실패 지점은
> 자동 계측만으로 거의 다 보인다. trace context(`traceparent` 헤더)가 서비스 경계를 자동으로 넘으므로,
> 10개 서비스를 거치는 요청도 하나의 trace로 꿰어진다.

### 2층 — 공통 횡단 계측 (사내 starter로 한 번만)

자동 계측이 못 잡는 것 중 **모든 서비스가 공유하는 패턴**을 처리. 사내 Spring Boot Starter가 이 층.

들어가는 것:

- 인프라 설정: OTLP 엔드포인트, 샘플링 정책, exporter 구성
- 공통 태그: `env`, `region`, `team`, `version` 등을 `ObservationFilter`로 **전 서비스에 자동 부착**
- 회사 고유의 공통 관심사: 여러 서비스가 똑같이 수행하는 외부 호출/인증 연계 등의 계측 로직을
  `ObservationConvention`으로 담아 재사용

태그 키·cardinality·마스킹 규칙이 바뀌어도 **starter만 고치면 전체에 반영**된다.
표준화와 거버넌스가 여기서 걸린다: "우리 회사 모든 서비스는 이 태그를 이 cardinality로 붙인다"를 강제.

### 3층 — 도메인 계측 (각 서비스가 자기 것만)

자동으로도 안 잡히고 공통도 아닌, **그 서비스만 아는 비즈니스 의미.**
실습한 `@ObservationKeyValue`가 정확히 이 층.

- `order` 서비스의 `order.tier`, `payment`의 `payment.method` — 각자 자기 도메인 태그를 자기 코드에 단다.
- 단순히 파라미터를 태그로 → **어노테이션**.
- 태그 로직이 복잡하거나 메서드 내부 값에 의존 → **convention 직접 호출**.
- 한 서비스 안에서도 둘을 섞어 쓴다.

### 3층 요약표

| 층 | 무엇 | 방법 | 책임 주체 |
|----|------|------|-----------|
| **1층 자동** | HTTP/DB/Kafka 등 표준 구간 | 내장 instrumentation / OTel Agent | 아무도 (자동) |
| **2층 공통** | 인프라 설정·공통 태그·회사 공통 호출 | 사내 starter + ObservationFilter/convention | 플랫폼팀 (1회) |
| **3층 도메인** | 서비스 고유 비즈니스 태그 | `@ObservationKeyValue` / convention | 각 서비스팀 |

## 2. 수집·운영 파이프라인

계측해서 신호를 만드는 것과, 그 신호를 모아 안정적으로 운영하는 것은 별개다.

### 2-1. Collector를 끼운다 (직접 전송 금지)

각 서비스가 백엔드(Tempo 등)로 직접 쏘지 않고, 중간에 **OpenTelemetry Collector**를 둔다.

```
[서비스 A] ─┐
[서비스 B] ─┼─→ [OTel Collector] ─→ Tempo / Prometheus / Loki (또는 Datadog 등)
[서비스 C] ─┘        │
                     └ 배칭 · 재시도 · 라우팅 · 민감정보 필터링 · tail sampling
```

장점: 백엔드 교체(Tempo→Datadog)나 멀티 전송 시 **서비스 코드 무변경**. 버퍼링·배칭으로 부하 흡수.
중앙에서 인증/egress 통제. MSA에서 사실상 표준 구성이다.

### 2-2. 샘플링은 tail sampling으로

- **head sampling** (요청 시작 시 10% 결정)의 문제: 에러나 느린 요청을 그냥 놓친다.
  → "방금 그 느린 요청"의 trace가 정작 안 남는, 트레이싱 도입 초기 최대 함정.
- **tail sampling** (Collector가 trace 종료 후 판단): "에러 났거나 / p99 넘게 느렸거나 / 특정 조건이면
  무조건 샘플링." 평상시 정상 요청은 1%만, 문제 있는 건 다 잡는다.

### 2-3. 메트릭과 trace의 역할 분담

| | 메트릭 (Prometheus/Mimir) | trace (Tempo) |
|---|---|---|
| 수집 범위 | 전 요청, 항상 | 샘플링됨 |
| 태그 | LOW cardinality만 | LOW/HIGH 전부 |
| 용도 | "뭔가 터졌다" **감지** | "구체적으로 뭔지" **파고듦** |
| 알림 | **여기 건다** | 안 건다 |

> 실습에서 `amount`를 그대로 안 넣고 HIGH/NORMAL 버킷(`order.tier`)으로 접은 이유가 이것:
> 메트릭에서 등급별 집계·비교·알림을 하기 위함. 시계열이 안 터지면서 깔끔하게 갈린다.

### 2-4. 전파(propagation)의 함정 관리

trace context가 서비스를 넘으려면 `traceparent` 헤더가 보존돼야 한다. 실무에서 시간을 가장 많이
잡아먹는 부분:

- API Gateway / LB가 `traceparent` 헤더를 떨구는 경우 → 헤더 보존 확인
- `@Async` / 스레드풀 / 메시지큐 경계에서 context 단절 → context propagation 설정,
  메시지 헤더에 trace context 싣기 (Kafka는 record header로 자동 전파되는 경우가 많음)

---

## 3. 실무 워크플로 (장애 대응 흐름)

```
1. 메트릭 알림     →  "express 주문 에러율 5% 초과" (메트릭, 항상 수집)
2. 태그로 범위 좁힘 →  Grafana에서 order.type / order.tier 별로 분해, 패턴 파악
3. 대표 trace 분석 →  Tempo에서 해당 시간대 trace 1건 열어 span 트리로 병목 구간 식별
4. 로그 점프       →  같은 trace ID로 Loki의 정확한 로그 라인으로 이동, 원인 확인
```

메트릭은 **넓고 얕게 항상**, trace는 **좁고 깊게 필요할 때**. 이 둘이 같은 계측(`@ObservationKeyValue` 등)에서
동시에 나오는 것이 Observation API를 쓰는 이유다.

## 4. 결론

"MSA에서 이 계측을 어떻게 처리하는가"의 답은
**어노테이션을 10개 서비스에 어떻게 안 흩뿌리지가 아니라, 흩뿌릴 것(도메인)과 모을 것(인프라·공통)을
나누는 계층 설계**다.

- 실습한 **어노테이션은 3층(도메인) 도구**다.
- MSA의 진짜 무게중심은 **1층(자동 계측 + Agent)** 과 **2층(사내 starter + Collector)** 에 있다.
- 수집은 Collector로 모으고, 샘플링은 tail로, 감지는 메트릭으로, 원인 추적은 trace로.

### 연계공통기반 / 플랫폼 포지션 관점 메모

산하 서비스들에 옵저빌리티 표준을 내려주는 위치라면 **2층을 만드는 쪽**에 가깝다.

- 산하 서비스가 공통으로 쓸 **observability starter**(OTel 설정·공통 태그·공통 연계 호출 convention)와
  **Collector 표준**을 내려주는 역할.
- 여러 시스템·외부 연계를 거치는 구조일수록 trace의 가치가 커진다(단일 로그로는 구간별 지연·실패가 안 보임).
- 폐쇄망에서는 Grafana 스택(LGTM)·Collector를 **내부에 직접 구축**해야 하는 제약이 따라붙는다.
- 설계 시 판단 포인트: 무엇을 starter 기본값으로 강제하고(거버넌스), 무엇을 각 팀 재량으로 열어둘지(자율).

# Hash 총정리

이 문서는 **Hash 기초 → HashMap 내부 구조 → 분산 시스템 활용 →
ConcurrentHashMap 동시성 문제**까지 백엔드 개발자 면접에서 자주 나오는
내용을 중심으로 정리한 문서입니다.

------------------------------------------------------------------------

# 1. Hash란 무엇인가

Hash는 **임의 길이의 데이터를 고정 길이 값으로 변환하는 함수** 또는 그
결과값입니다.

    Input Data → Hash Function → Fixed Length Hash Value

예

    "hello"
    → 5d41402abc4b2a76b9719d911017c592

특징

-   입력 길이 : 가변
-   출력 길이 : 고정
-   같은 입력 → 항상 같은 출력
-   입력이 조금만 바뀌어도 결과가 크게 달라짐

이러한 특징 때문에 Hash는 다음 분야에서 사용됩니다.

-   자료구조 (HashMap, HashSet)
-   비밀번호 저장
-   데이터 무결성 검증
-   캐시 시스템
-   분산 시스템

------------------------------------------------------------------------

# 2. Hash의 핵심 특징

## 2.1 Deterministic

같은 입력은 항상 같은 Hash 값을 생성합니다.

    hash("A") = 123
    hash("A") = 123

------------------------------------------------------------------------

## 2.2 Avalanche Effect

입력이 조금만 바뀌어도 결과는 완전히 달라집니다.

    hash("hello")
    hash("hello!")

출력은 전혀 다른 값

이 특성은 **보안 / 분산 시스템**에서 중요합니다.

------------------------------------------------------------------------

## 2.3 Collision

다른 입력이 같은 Hash 값을 가질 수 있습니다.

    hash(A) = 123
    hash(B) = 123

이를 **Hash Collision**이라고 합니다.

충돌을 완전히 없앨 수는 없기 때문에 자료구조에서는 **충돌 해결 전략**이
필요합니다.

------------------------------------------------------------------------

# 3. HashMap 동작 원리

HashMap은 내부적으로 다음 구조를 가집니다.

    Array (bucket)
      |
      +-- Node
      +-- Node
      +-- Node

즉

    HashMap = Array + Linked Structure

------------------------------------------------------------------------

# 4. HashMap 데이터 저장 과정

예

    map.put("apple", 1)

동작 과정

### 1단계 hashCode 계산

    key.hashCode()

예

    "apple".hashCode()

------------------------------------------------------------------------

### 2단계 Hash 보정

Hash 충돌을 줄이기 위해 추가 연산 수행

    hash = h ^ (h >>> 16)

------------------------------------------------------------------------

### 3단계 Bucket Index 계산

    index = (n - 1) & hash

예

    capacity = 16
    index = hash & 15

------------------------------------------------------------------------

### 4단계 Bucket에 Node 저장

    bucket[3]

    Node(key,value)

------------------------------------------------------------------------

# 5. Hash Collision 해결 방법

## 5.1 Chaining

같은 bucket에 LinkedList 형태로 저장

    bucket

    A → B → C

시간복잡도

    O(n)

------------------------------------------------------------------------

## 5.2 Tree 구조 (JDK8)

충돌이 많아지면

    LinkedList → Red Black Tree

시간복잡도

    O(log n)

Tree 전환 조건

    bucket size >= 8
    table size >= 64

------------------------------------------------------------------------

# 6. 좋은 Hash 함수 특징 (실무)

좋은 Hash 함수는 다음 특성을 가져야 합니다.

### 1. 균등 분포

데이터가 bucket에 고르게 분산되어야 합니다.

### 2. Avalanche Effect

입력 변화 → 큰 출력 변화

### 3. 빠른 계산 속도

Hash 계산은 매우 빈번하게 발생합니다.

예

-   HashMap 조회
-   Redis Key
-   Kafka Partition

### 4. 낮은 충돌률

충돌이 많으면 성능이 급격히 떨어집니다.

------------------------------------------------------------------------

# 7. equals()와 hashCode()

HashMap은 두 단계로 데이터를 찾습니다.

    1단계 hashCode
    2단계 equals

과정

    hashCode → bucket 찾기
    equals → 실제 key 비교

Java 규칙

    equals() == true
    → hashCode() 반드시 동일

이 규칙을 지키지 않으면 HashMap에서 데이터를 찾을 수 없는 문제가
발생합니다.

------------------------------------------------------------------------

# 8. HashMap capacity가 2의 거듭제곱인 이유

HashMap은 index 계산을 다음과 같이 합니다.

    index = (n - 1) & hash

이 연산은

    hash % n

과 동일하지만

    % 연산보다 훨씬 빠릅니다.

따라서 capacity는 항상

    16
    32
    64
    128

처럼 **2의 거듭제곱**을 사용합니다.

------------------------------------------------------------------------

# 9. HashMap Resize

Load Factor 기본값

    0.75

예

    capacity = 16
    threshold = 12

12개 초과 시 resize

    16 → 32 → 64

# 15. ConcurrentHashMap size()가 부정확한 이유

동시성 환경

    Thread A put
    Thread B remove
    Thread C size

size 계산 중에도 데이터가 변경됩니다.

따라서 Java는

    weak consistency

전략 사용

구조

    baseCount
    counterCells[]

size 계산

    baseCount + sum(counterCells)

동시에 값이 변할 수 있기 때문에 **약간의 오차 가능**

# 17. 설명

## HashMap이 O(1)인 이유

> HashMap은 key의 hashCode를 이용해 bucket index를 바로 계산하기 때문에
> 평균적으로 O(1)의 시간 복잡도를 가집니다.

------------------------------------------------------------------------

## equals와 hashCode를 같이 override 해야 하는 이유

> HashMap은 먼저 hashCode로 bucket을 찾고 같은 bucket 내에서 equals로
> key를 비교합니다. 따라서 equals가 true라면 hashCode도 반드시 같아야
> 합니다.

------------------------------------------------------------------------

## HashMap capacity가 2의 거듭제곱인 이유


> HashMap은 bucket index 계산을 `(n-1) & hash` 비트 연산으로 수행합니다.
> 이 방식은 `%` 연산보다 빠르며 capacity가 2의 거듭제곱일 때만 균등한
> 분포가 보장됩니다.

------------------------------------------------------------------------

## ConcurrentHashMap size()가 정확하지 않은 이유

> ConcurrentHashMap은 높은 동시성을 위해 카운터를 여러 셀로 분산하는
> 구조를 사용합니다. size 계산 중에도 다른 스레드가 데이터를 변경할 수
> 있기 때문에 약간의 오차가 발생할 수 있으며 이는 성능을 위한 weak
> consistency 설계입니다.

------------------------------------------------------------------------

# 18. 핵심 요약

Hash 핵심 정리

1.  Hash는 데이터를 고정 길이 값으로 변환하는 함수
2.  HashMap 평균 시간복잡도 O(1)
3.  충돌 해결 → LinkedList / Tree
4.  capacity는 2의 거듭제곱
5.  JDK8 resize 최적화
6.  HashMap은 thread-safe 아님
7.  멀티스레드 환경 → ConcurrentHashMap 사용
8.  ConcurrentHashMap은 weak consistency 사용
